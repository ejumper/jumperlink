Making the Site

In my other posts I have gone through my step by step process, however, due to the enormity of the work and the constant revising this project required it just didn't seem practical to document and screenshot every step I took, so I will instead have a simple run through of how the site works.

From the start I had the idea to keep the website on a single webpage with a container that I could inject other HTML files into to display whatever I wanted. I also wanted to design the site so that it would work equally well on mobile and desktop. This was probably not a pragmatic approach as creating a scaling that worked at any size was by far the hardest part but it also resulted in me having to gain a much better understanding, of the declarations I was using where I otherwise could have just set...

The three main elements, the title bar, navigation menu and content container are placed using a flex-box.

The items in the title bar are placed using absolute positioning.

I created a clamp to scale my name up and down and that clamp was used as the basis for all the scaling on the page.

The mail and phone icons are scaled to a fraction of that clamp and the "view resume" button is scaled to equal the sum of the icons width and the gap between them to maintain symmetry.

The navigation menu is a simple flexbox with :hover and :select affects in CSS to create the animations. It is scaled by scaling the side margins with the size of the viewport. This was probably not the best solution because as the viewport widens the menu scales up as it should but then eventually shrinks. The issue is that because of my cascading use of variables everything else on the page would also have to be resized and it is not enough of a problem to warrant the effort.

I created HTML files for each of the menu items that JavaScript injects into the content box.

For the Home Labs I listed all of my projects in an HTML file and used CSS to turn it into a grid. Then I created more HTML files for each specific project which are injected into the content box when a project is selected from the home labs tab.

I plan to create a back button in the content box that appears when you open a home lab project to make navigating back to the home lab grid menu more intuitive, but at the time of writing I have figured out a workable solution and selecting the home lab button again accomplishes this anyway.

Deploying

Cloudflare Pages has two options for deployment: Importing Git repositories and direct uploading. I'll be using Git for easy version control and deployment.

I will of course be using GitHub to hold my Git repository. And the first step is to Create a new repository

I choose my repository name and uncheck add a README, select Create & copy the repository's URL

Now in Terminal I cd to my project folder, use git init to to create a new Git repository, then git remote add origin and my repository url to link my local project files to my repository, then (after remembering to add git) use git add . to stage the changes in my working directory, then I commit the changes with git commit -m

I then rename the branch to "main" and use git push -u origin main to set the main branch as the default branch for future commits.

I quickly check that my files have appeared in my GitHub repo then reopen cloudflare to create Cloudflare Page

I connect my GitHub account, add my account and select my repository.
